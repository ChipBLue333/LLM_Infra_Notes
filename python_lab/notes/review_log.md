## Problem 001: Two Sum

### 题型
Array / Hash Map

### 我的错误
- 基础实现使用了 $O(n^2)$ 的暴力解法，虽然结果正确，但没有利用哈希表进行优化。

### 正确思路
- 遍历 `nums`。
- 对每个 `num`，检查 `target - num` 是否已经出现在哈希表中。
- 如果出现，直接返回对应下标和当前下标。
- 如果没有，将 `num` 作为键，其下标作为值，存入哈希表。

### 关键模式
- **哈希表（Hash Map）查询优化**：利用空间换时间，将查询的时间复杂度从 $O(n)$ 降为 $O(1)$。

### 复杂度
- Time: $O(n)$ （优化后）
- Space: $O(n)$ （优化后）

### 下次注意
- 当涉及到“在数组中查找配对元素”时，优先考虑使用字典（哈希表）来记录已经扫描过的元素。

## Problem 002: Valid Parentheses

### 题型
String / Stack / Hash Map

### 我的错误
- (无严重错误，一次性 pass)
- 风格瑕疵：使用了 Python 内置关键字 `str` 作为变量名；字典在循环内部重复定义导致不必要的开销；不需要索引时使用了 `enumerate`。

### 正确思路
- 使用列表 `my_list` 模拟栈（Stack）。
- 遇到左括号，入栈。
- 遇到右括号，检查栈是否为空，不为空则弹出栈顶元素。
- 使用字典 `seen_map` 验证弹出的左括号和当前的右括号是否真正成对。
- 全部遍历完后，确认栈是否已经清空。

### 关键模式
- **栈结构 (Stack)**: FILO (先进后出)，由于括号嵌套必然是从最内层开始闭合，非常适合用栈的特性去消除配对。

### 复杂度
- Time: $O(n)$
- Space: $O(n)$ 

### 下次注意
- 不要使用 Python 的保留字 / 内置类型（如 `str`, `list`, `dict`, `int`）作为变量名。
- 常量字典或者不变的映射关系，应当定义在循环外面初始化，避免在循环体内反复创建损耗性能。

## Problem 003: Merge Sorted Array

### 题型
Array / Two Pointers (In-place)

### 我的错误
- (在教练提示下完成，伪代码转换成代码非常准确，没有逻辑错误)

### 正确思路
- 使用三个指针：`p1` 指向 `nums1` 真实末尾，`p2` 指向 `nums2` 末尾，`p` 指向 `nums1` 容量末尾。
- 从后往前比较 `nums1[p1]` 和 `nums2[p2]`，把较大的值填入 `nums1[p]`，然后对应指针往前移。
- 考虑到边界情况：如果 `p1` 先走完，只要把 `nums2` 剩下的数字依次填入最前面即可；如果 `p2` 先走完，说明 `nums1` 剩下的数字本身就有序且在正确位置，无需处理。

### 关键模式
- **双指针 (逆向思维) / Two Pointers (Backward)**: 当需要原地修改且尾部有足够空间时，从后往前填坑可以避免数据被提前覆盖。

### 复杂度
- Time: $O(m + n)$
- Space: $O(1)$

### 下次注意
- 原地修改 (In-place) 时，如果“从前往后”会导致数据被覆盖或产生高昂的移动成本，考虑“从后往前”。

## Problem 004: Best Time to Buy and Sell Stock

### 题型
Array / Dynamic Programming

### 我的错误
- 最初尝试用双层循环和状态标识位来模拟，逻辑复杂且漏掉了正确答案。
- 这之后改用了一层循环，但内部使用了切片 `prices[0:i]` 加 `min()`，导致了 $O(N^2)$ 的隐藏时间复杂度和 $O(N)$ 的由于切片导致的空间复杂度。

### 正确思路
- 只需要一次遍历（One Pass）。
- 动态维护一个 `min_buy_price` 记录迄今为止见过的最低价格。
- 每次用当前价格减去最低价格，尝试更新最大利润。

### 关键模式
- **动态维护历史极值**：不需要反复回头看历史数据，而是随着遍历一天天更新历史状态。

### 复杂度
Time: O(N)
Space: O(1)

### 下次注意
- 注意时间复杂度的隐性退化（不要在 for 循环里用耗时的 python 切片和内置函数 `min()`、`max()` 去扫描长列表）。
- 局部变量不能与函数名冲突（shadowing）。

## Problem 005: Longest Substring Without Repeating Characters

### 题型
Hash Map / Sliding Window / String

### 我的表现
一次性写出最优解！代码一次过，没有任何逻辑漏洞。

### 正确思路
- 使用滑动窗口（双指针 `left` 和 `right`）。
- 使用字典 `char_index` 记录字符最后一次出现的位置，实现 O(1) 的查找和左指针的跳跃。
- 如果遇到的字符属于历史重复字符，并且还在当前窗口内（`char_index[s[right]] >= left`），说明窗口内产生了冲突，直接将 `left` 跨步移动跃过冲突位置。
- 每次移动后更新字符的最新位置，并计算最大长度。

### 关键模式
- **Sliding Window (优化版)**：通常滑动窗口遇到不满足条件时需要使用 `while` 循环让 `left` 一步步向右走直到满足条件。借用哈希表记录位置，可以让 `left` 一步跳跃到位，将时间复杂度严格控制在 $O(N)$ 且常数更小。

### 复杂度
Time: O(N) 
Space: O(Σ) 其中 Σ 表示字符集大小，对于 ASCII 最大为 128 或 256。

## Problem 007: Minimum Size Subarray Sum

### 题型
Array / Sliding Window

### 我的错误
- 无错误，顺利实现了滑动窗口模式。

### 正确思路
- 使用滑动窗口，右指针不断向右扩张，扩展窗口和。
- 一旦当前窗口和大于等于 target，尝试通过向右移动左指针来缩窄窗口，并在缩窄的过程中更新最小窗口长度。
- 注意指针移动和长度计算的顺序匹配（长度为 `right - left` 是因为 right 提前进了一步）。

### 复杂度
- Time: O(n)
- Space: O(1)

### 下次注意
- Python中可以用 `float('inf')` 来初始化求最小值的变量，比用具体常数（如 `10**5 + 1`）更清晰、也更有扩展性。

## Problem 008: Search Insert Position

### 题型
Array / Binary Search

### 我的表现
- 一次性写出标准二分查找解法，可见测试全部通过。
- 能正确处理 target 存在、应插入中间、应插入头部、应插入尾部、数组长度为 1 等边界情况。

### 正确思路
- 使用 `left` 和 `right` 表示当前搜索区间。
- 每次计算 `mid`，比较 `nums[mid]` 和 `target`。
- 如果找到目标值，直接返回 `mid`。
- 如果目标值更大，说明插入位置在右半部分，移动 `left = mid + 1`。
- 如果目标值更小，说明插入位置在左半部分，移动 `right = mid - 1`。
- 循环结束后，`left` 就是第一个大于等于 `target` 的位置，也就是插入位置。

### 复杂度
- Time: O(log n)
- Space: O(1)

### 下次注意
- 二分题的关键是明确循环结束后指针的含义。本题中 `left` 最终代表插入位置。
- Python 整数不会溢出，但 `mid = left + (right - left) // 2` 是很好的通用写法，值得保留。

## Problem 009: Reverse Linked List

### 题型
Linked List / Pointer

### 我的表现
- 在提示后能正确写出迭代版链表反转。
- 对 `previous`、`current`、`next_node` 三个变量的职责理解到位。
- 可见测试全部通过，包括空链表、单节点链表、双节点链表和多节点链表。

### 正确思路
- 使用 `previous` 表示已经反转好的前半段。
- 使用 `current` 表示当前正在处理的节点。
- 每一轮先保存 `next_node = current.next`，避免断链后丢失后续节点。
- 将 `current.next` 指向 `previous`，完成当前节点的反转。
- 再依次移动 `previous = current`、`current = next_node`。
- 当 `current` 变成 `None` 时，`previous` 就是新的头节点。

### 复杂度
- Time: O(n)
- Space: O(1)

### 下次注意
- 链表题里，修改 `next` 之前先保存后路，这是防止丢节点的关键习惯。
- 可以保留有解释力的注释，但熟练后可以减少逐行注释，让核心步骤自己说话。

## Problem 011: Binary Tree Level Order Traversal

### 题型
Tree Traversal / BFS / Queue

### 我的情况
- 充分理解了 `collections.deque` 的应用。
- 一次性写出了带有内层 size 控制的经典 BFS 层序遍历。
- 对 Python 中 `for _ in range(size)` 的占位符惯用法提出疑问。

### 正确思路
- **BFS 层序模板**：在每次外层 `while queue:` 开始时，先取得当前队列长度 `size = len(queue)`。由于队尾会不断追加下一层的节点，利用一个 `for \_ in range(size)` 的内循环，精准地只从队头弹出当前层的节点。这样就能漂亮地把树拆分成一层一层的结构。

### 关键模式
- 利用 `deque` 做到 $O(1)$ 的出队（`popleft`）。
- **通过提前记录 size 来截断队列的增长，是处理所有涉及“按层/按步”BFS问题的通用解法。**

### 复杂度
Time: O(N)
Space: O(N)

### 下次注意
- `_` 是常用的 throwaway 变量，表示只需要循环次数，不需要具体序号时使用。

## Problem 012: Maximum Depth of Binary Tree

### 题型
Tree Traversal / DFS

### 我的情况
- 选用了递归 (DFS) 的解法。
- 完美理解了“大问题转化为小问题”的递归思维。
- 一次性写对，结构非常清晰。

### 正确思路
- **递归版**：当前树的最大深度 = `max(左子树最大深度, 右子树最大深度) + 1`。边界条件是空树返回 0。

### 关键模式
- 这是树形 DP (Dynamic Programming on Trees) 或后序遍历（Bottom-up）求信息的最基础模板。很多类似求树高、树直径、判断平衡树的问题，都是核心沿用这套逻辑。

### 复杂度
- Time: O(N)
- Space: O(H) (H 为树的高度，最坏 O(N))

### 下次注意
- 代码极其精简干净，继续保持这种思维清晰度！

## Problem 013: Same Tree

### 题型
Tree Traversal / DFS / Recursion

### 我的情况
- 一次性写出正确解法，可见测试全部通过。
- 能正确区分“两棵树都为空”和“只有一棵树为空”这两个关键边界条件。
- 对两棵树同步递归的结构理解到位。

### 正确思路
- 如果 `p` 和 `q` 都是空节点，说明当前子树相同。
- 如果 `p` 和 `q` 只有一个为空，说明结构不同。
- 如果两个节点都存在但值不同，说明节点值不同。
- 当前节点通过后，再递归比较左子树和右子树，两边都相同才返回 `True`。

### 关键模式
- **双树同步递归**：处理两棵树时，每一层都同时检查两个节点的存在性、值，以及对应方向的子树。
- 这是判断相同树、对称树、子树匹配等问题的基础模式。

### 复杂度
- Time: O(N)
- Space: O(H) (H 为树的高度，最坏 O(N))

### 下次注意
- 继续保持先处理空节点边界，再处理当前节点，再递归子问题的顺序。

## Problem 014: Symmetric Tree

### 题型
Tree Traversal / DFS / Recursion

### 我的情况
- 在提示后写出了标准的镜像递归解法。
- 可见测试全部通过。
- 能正确区分 Same Tree 的同方向比较和 Symmetric Tree 的交叉方向比较。

### 正确思路
- 空树是对称树。
- 对非空树，判断 `root.left` 和 `root.right` 是否互为镜像。
- 辅助函数 `is_mirror(left, right)` 中先处理空节点边界：
  - 两个都为空，返回 `True`。
  - 只有一个为空，返回 `False`。
  - 两个节点值不同，返回 `False`。
- 当前节点通过后，递归比较 `left.left` 和 `right.right`，以及 `left.right` 和 `right.left`。

### 关键模式
- **镜像递归 (Mirror Recursion)**：比较两棵树时，不是同方向比较，而是交叉比较。
- 与 Same Tree 的区别：
  - Same Tree: `p.left` vs `q.left`，`p.right` vs `q.right`
  - Symmetric Tree: `left.left` vs `right.right`，`left.right` vs `right.left`

### 复杂度
- Time: O(N)
- Space: O(H) (H 为树的高度，最坏 O(N))

### 下次注意
- 看到“对称”“镜像”“翻转后相同”这类关键词，要第一时间想到交叉递归。

## Problem 015: Path Sum

### 题型
Tree Traversal / DFS / Recursion

### 我的情况
- 一次性写出正确解法，可见测试全部通过。
- 能正确理解“路径必须从根节点到叶子节点”，没有把中间节点误判为合法路径。
- 当前实现先递归左右子树，再在叶子节点处覆盖结果，逻辑正确但控制流略绕。

### 正确思路
- 空树返回 `False`。
- 每到一个节点，就用 `target_sum - root.val` 表示剩余还需要凑出的路径和。
- 如果当前节点是叶子节点，只需要判断 `root.val == target_sum`。
- 如果不是叶子节点，则递归检查左子树或右子树是否存在合法路径。

### 关键模式
- **根到叶路径递归**：递归时携带“剩余目标值”或“当前累计和”。
- 叶子节点是判断答案的关键位置，不能在中间节点提前返回成功。

### 复杂度
- Time: O(N)
- Space: O(H) (H 为树的高度，最坏 O(N))

### 下次注意
- 树递归题里，如果叶子节点是答案判断点，可以先判断叶子，再决定是否继续递归，这样控制流更直接。

## Problem 016: Invert Binary Tree

### 题型
Tree Traversal / DFS / Recursion

### 我的情况
- 一次性写出正确解法，可见测试全部通过。
- 能正确处理空树、单节点、只有左孩子、只有右孩子和完整二叉树等情况。
- 当前写法采用“先递归翻转左右子树，再交换当前节点左右指针”的后序思路，逻辑正确。

### 正确思路
- 空树直接返回 `None`。
- 对每个非空节点，交换它的左孩子和右孩子。
- 然后继续递归翻转交换后的左右子树，或者先递归左右子树再交换当前节点，两种顺序都可以。
- 最后返回原来的 `root`，因为这是原地修改二叉树。

### 关键模式
- **二叉树原地修改递归**：函数既要修改当前节点的指针，也要把同样的操作递归应用到左右子树。
- 这类题的返回值通常还是当前节点本身，用来保持递归调用能够接回父节点。

### 复杂度
- Time: O(N)
- Space: O(H) (H 为树的高度，最坏 O(N))

### 下次注意
- 交换两个变量时可以直接使用 Python 的 tuple unpacking：`root.left, root.right = root.right, root.left`。
- 如果代码已经清楚，避免多余的临时初始化，例如 `temp = None` 后立刻覆盖就没有必要。

## Problem 017: Binary Tree Paths

### 题型
Tree Traversal / DFS / Recursion

### 我的情况
- 一次性写出正确解法，可见测试全部通过。
- 能正确把递归返回值设计成“当前子树所有根到叶路径的字符串列表”。
- 能正确处理空树、单节点、单侧链、负数节点和左右子树都有路径的情况。

### 正确思路
- 空树返回空列表。
- 叶子节点返回只包含当前节点值的列表，例如 `["1"]`。
- 对非叶子节点，分别递归获取左子树路径和右子树路径。
- 把当前节点值作为前缀，加到每条子路径前面，形成完整路径。

### 关键模式
- **递归返回列表并由父节点加工**：子问题只负责返回自己的结果，父节点负责把当前层信息拼接进去。
- 这是树上收集答案类 DFS 的基础模式，后面做 Path Sum II、所有根到叶路径、回溯收集路径都会继续用到。

### 复杂度
- Time: O(N * H) 或 O(total_path_string_length)，因为每条路径字符串拼接需要复制字符。
- Space: O(N * H) 用于结果字符串，递归栈为 O(H)。

### 下次注意
- `left_paths + right_paths` 会创建一个额外列表。数据量小时无所谓，但可以分别遍历左右列表来避免这一步。
- 注释要解释“为什么这么做”，不要重复代码已经表达清楚的内容。

## Problem 018: Path Sum II

### 题型
Tree Traversal / DFS / Recursion / Backtracking

### 我的情况
- 一次性写出正确解法，可见测试全部通过。
- 复用了 Binary Tree Paths 的“子树返回路径列表”模型，并加入了剩余目标值 `target_sum - root.val`。
- 能正确处理空树、单节点命中、单节点不命中、负数路径、多条合法路径和无合法路径。

### 正确思路
- 空树返回空列表。
- 如果当前节点是叶子节点，只有当 `root.val == target_sum` 时才返回 `[[root.val]]`。
- 如果当前节点不是叶子节点，递归到左右子树时把目标值减去当前节点值。
- 子树返回满足剩余目标值的路径后，当前节点把自己的值加到每条路径前面。

### 关键模式
- **剩余目标值递归**：递归时把目标值逐层扣掉，到叶子节点时判断当前节点是否刚好等于剩余目标值。
- **递归返回路径列表**：子树负责返回合法路径，父节点负责给路径加前缀。
- 这题也可以用回溯写法：维护一个共享 `path`，进入节点时 append，离开节点时 pop。

### 复杂度
- Time: O(N * H) 或 O(total_path_length)，因为构造路径列表时会复制路径。
- Space: O(N * H) 用于结果和中间路径，递归栈为 O(H)。

### 下次注意
- `left_paths + right_paths` 会创建额外列表，可以分别遍历左右路径。
- `[root.val] + path` 每次都会创建新列表，这是正确但有额外复制成本的写法；回溯写法能减少中间列表创建，但要小心 append/pop 成对出现。

## Problem 019: Subsets

### 题型
Backtracking / DFS / Array

### 我的情况
- 写出了标准回溯解法，可见测试全部通过。
- 能正确使用 `start` 控制下一层搜索范围，避免重复选择和逆序组合。
- 能正确使用 `path.copy()` 保存当前路径，避免结果列表被后续 `path.pop()` 污染。

- 使用 `result` 保存所有子集，使用 `path` 保存当前正在构造的子集。
- 每进入一层递归，先把当前 `path` 的副本加入结果。
- 从 `start` 开始遍历候选元素：
  - 选择 `nums[i]`；
  - 递归搜索 `i + 1` 之后的元素；
  - 回溯撤销这次选择。

### 关键模式
- **回溯三步**：做选择、递归、撤销选择。
- **组合型搜索的 start 下标**：下一层只能从当前元素后面继续选，避免重复组合。
- **保存路径时复制**：结果里应保存当前路径的快照，而不是共享的 `path` 对象。

### 复杂度
- Time: O(N * 2^N)
- Space: O(N * 2^N) 用于结果，递归栈和 path 为 O(N)。

## Problem 020: Permutations

### 题型
Backtracking / DFS / Array

### 我的情况
- 写出了标准的全排列回溯解法，可见测试全部通过。
- 能正确使用 `used` 数组记录当前路径中已经使用过的元素。
- 能在 `len(path) == len(nums)` 时保存完整排列，并及时 `return`。
- 能保持 `path.append()` / `path.pop()` 与 `used[i] = True` / `used[i] = False` 成对出现。

### 正确思路
- 使用 `result` 保存所有完整排列，使用 `path` 保存当前排列前缀。
- 使用 `used` 标记每个下标对应的元素是否已经在当前 `path` 中。
- 每一层都从头遍历所有下标：
  - 如果 `used[i]` 为 `True`，跳过；
  - 否则选择 `nums[i]`，继续递归；
  - 递归返回后撤销选择和使用标记。
- 当 `path` 长度等于 `nums` 长度时，说明一个排列完成。

### 关键模式
- **排列型回溯使用 used**：每一层都可以从全部元素中选择，但同一个路径里不能重复使用同一元素。
- **组合 / 子集使用 start**：下一层只从当前位置之后继续选，避免重复组合。
- **状态成对恢复**：修改了几个状态，回溯时就要恢复几个状态。

### 复杂度
- Time: O(N * N!)
- Space: O(N * N!) 用于结果，递归栈、path 和 used 为 O(N)。

### 下次注意
- 本题约束 `len(nums) >= 1`，所以空数组提前返回不是必要逻辑；如果保留也没问题。
- 函数之间按 PEP 8 建议空两行，例如 `return result` 和 `def normalize` 之间。
